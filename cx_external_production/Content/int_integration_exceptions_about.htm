About Integration Exceptions

The Integration Exceptions tool is used to view and resubmit failed integration requests. There are many different types of integration exceptions that can occur. This topic focuses on the two most common: digital rights and fulfillment exceptions.

When an integration exception occurs for a line item in an order, the line item is placed on hold and will not be fulfilled until the exception is fixed. Any additional line items sent to the same fulfiller for that order will most likely be placed on hold as well. The exception to this rule is when the fulfiller has been expressly told that partial shipments are allowed. The shopper cannot be billed for any line items that have not yet been shipped, therefore revenue for these line items cannot be recognized. If left unfixed, both Digital River and the client lose money. To release the line item, the exception must be fixed and the transaction re-extracted. If the problem cannot be fixed, the line item or transaction can be canceled.

Warning: Once a line item (product) or order (transaction) is canceled, it cannot be un-canceled. You should only cancel line items or order when you are sure the exception cannot be fixed.

Integration Exception Report

Each day, an integration exception report is sent via email to the Store Operations team. This email contains a list of the integration exceptions that occurred for all sites, with the Site IDs and other relevant information for the specific issues, listed in a table. The email subject line also contains the Site IDs of sites reported on in the email. If no action is taken to fix the exceptions, the line items remain in limbo, and Digital River and the client cannot recognize revenue on those items.

Note: You can create rules (or filters) in Microsoft® Outlook® so that only emails with specific Site IDs in the subject line appear in your Inbox. See the Help file in Outlook for more information about creating rules to filter your Inbox. For reference when creating the rule, the integration exception reports are sent from the following email address: PacificIntegrationServices@digitalriver.com

Digital Rights Exceptions

A digital rights request exception occurs when the digital rights for a product (serial number or unlock code) cannot be retrieved and given to the customer. This occurs when the digital rights have been depleted (the dispenser is empty) or when Digital River's integration with an external API fails or is rejected.

When you view the transaction details for a digital rights exception, you can see the specific reason for the exception listed for the line item(s) in question.

The following example shows theTransaction Details page for a digital rights exception. The “Transaction Exception” stated is “Empty Key Response.” This means that the digital rights for at least one product in the order are depleted or the external DRM source rejected the request because the product information submitted does not match what they have on file.

In the list of items for the transaction, the product with text in the Exception column is the product with the exception you must fix. You can click the Product ID to view the details of the product and from the product details, make changes to the Digital Rights, Part Number, SKU, or Manufacturer Part Number (whatever caused the exception). Once you make the changes and deploy the product, the exception should be fixed and you can re-extract all transaction(s) with that exception.

Tip: You can re-extract one or more transaction at the same time. If you fixed a problem that caused an exception on multiple transactions, you can select every transaction with the exception and re-extract. You do not have to re-extract each transaction separately.

If a third party is used to deliver digital rights and the integrated API failed, you can contact the supplier and get the correct DRM to issue to the customer. Once you do this, you would select the product on this screen and click Issue Manually. In the Manually Issue Digital Rights window that appears, you enter the serial number or unlock code for the product (that you get from the external DRM supplier)and click Issue. The digital rights information you enter will then be sent to the customer.

If the line item must be canceled because the exception cannot be fixed, you must enter text about the reason for the cancellation in the Corrective Action Taken or Removal Reason field. Then, you select the product and click Cancel. This should only be done when you cannot fix the problem that causes the exception. If the transaction contains additional line items without exceptions, you should re-extract the transaction after you cancel the problem line item. If the order contains only the one canceled line item or all line items within a given order are canceled, the entire order is canceled.

Pending Digital Rights or Failed Digital Rights

When a product has a pending digital rights or failed digital rights exception it means that the DRM for the product could not be sent or displayed to the customer. Pending digital rights exceptions most often occur because the external (non-Digital River) DRM dispenser for the product does not recognize the product information. Failed digital rights exceptions most often occur because the DRM dispenser (list of serial numbers and unlock codes) is empty.

When a product has a pending or failed digital rights exception it means one of the following has occurred:

  1. The Digital River hosted DRM dispenser has run out of serial numbers and/or unlock codes and needs to be replenished.
  2. There is a communication problem with the external API from which the serial numbers and/or unlock codes are retrieved.
  3. The external API from which the serial numbers and/or unlock codes are retrieved has rejected the request.

In case 1 above, you must contact the client and have them add serial numbers and/or unlock codes for the product (line item). Instructions for how to do this can be found through About the Digital Rights Option.

In case 2 above, the digital rights request will be retried for up to 21 days. If, after 21 days, the API still fails and the digital rights still have not been retrieved, the transaction will require manual intervention from Digital River. This involves contacting the client or customer (shopper) to determine what action should be taken. The DRM for the line item (product) may be issued manually, or the line item and/or order may be canceled.

In case 3 above, the cause or problem that appears for the exception on the Manage Integration Exceptions page must be fixed and the transaction re-extracted. In most cases, a digital rights request is rejected because the Part Number is not recognized by the API. You should verify the SKU, External Reference Number, and/or Fulfiller Part Number for the product (line item) causing the exception. If you make changes to the product make sure you deploy the product before re-extracting the transaction.

Fulfillment Exceptions

A fulfillment request exception occurs when one of the following occurs:

  1. There is a communication problem with the fulfiller's fulfillment request API.
  2. The fulfiller's API has rejected the fulfillment request. In most cases, this is due to the Part Number not being recognized or has been discontinued.

In case 1 above, the fulfillment request will be retried for up to 21 days. If, after 21 days, the API still fails and the fulfillment request cannot be completed, the transaction will require manual intervention from Digital River. This involves contacting the client or customer (shopper) to determine what action should be taken. The transaction (order) may have to be canceled.

In case 2 above, the cause of problem that appears for the exception on the Manage Integration Exceptions page must be fixed and the transaction re-extracted. For example, if the exception is a "Fulfiller Indicated SKU Error Event" it means that the Part Number is not recognized by the fulfiller. In this scenario, the SKU, External Reference Number and/or Fulfiller Part Number may need to be updated and the product redeployed before the transaction is re-extracted.

The following example shows what appears on the Transaction Details page for the fulfillment exception explained in case 2 above. The “Transaction Exception” stated is “Fulfiller Indicated SKU-Error Event.” This means that the fulfiller SKU for a product is not valid.

In the list of items for the transaction, the product with text in the Exception column is the product with the exception you must fix. You can click the Product ID to view the details of the product and from the product details, make changes to the fulfiller, fulfiller part number, warehouse, or warehouse part number (whatever caused the exception). Once you make the changes and deploy the product, the exception should be fixed and you can re-extract the transaction.

If the line item must be canceled because the exception cannot be fixed, you must enter text about the reason for the cancellation in the Corrective Action Taken or Removal Reason field. Then, you select the product and click Cancel. This should only be done when you cannot fix the problem that causes the exception. If the transaction contains additional line items without exceptions, you should re-extract the transaction after you cancel the problem line item.

Bad Part Number

When a line item has a "bad part number" exception, it means that the part number saved with the product does not match what the fulfiller has on record. Products with this exception are essentially "dead" products to the fulfiller. Most exceptions of this type require the product part number or fulfiller part number to be updated.

To fix bad part number exceptions, you must first determine the Fulfiller and Warehouse Location for the product. Then, you contact the Fulfiller to determine the correct Part Number (the one they have on record). Once you get the fulfiller part number, you edit the product and update the Fulfiller Part Number. After you deploy the product, you can re-extract the transaction.

Discontinued Part Number

When a line item has a "discontinued part number" exception, it means that the part number saved with the product has been discontinued by the fulfiller. This is usually because the fulfiller no longer stocks the product or because the product has been replaced with another product that has a different part number.

To fix discontinued part number exceptions, you must first check if the line item (product) has more than one fulfiller.

  • If the line item (product) is associated with more than one fulfiller, you must first disassociate the fulfiller that had the exception from the product.

    For example, if the product has multiple fulfillers but one (DSS) says the part number is discontinued, you would remove DSS as a fulfiller for the product, redeploy the product, and re-extract the transaction. The line item in the transaction will be sent to one of the remaining fulfillers associated with the product for fulfillment.
  • If the line item (product) is only associated with the fulfiller where the exception occurred, you must cancel the line item. If the transaction has additional line items (products) that do not have exceptions, you must re-extract the transaction. In this scenario, you may also want to retire the product since it cannot be fulfilled, or get fulfillment set up with another fulfiller, or get more inventory sent to the fulfiller.
